# oa-tools

[![Built with Ona](https://ona.com/build-with-ona.svg)](https://app.ona.com/#https://github.com/Interested-Deving-1896/oa-tools)

<!-- AI:start:what-it-does -->
This project provides tools for remastering and customizing Linux distributions in a universal manner, aiming to address the challenge of managing diverse system configurations. It is used by developers and system administrators to streamline the process of creating and maintaining tailored operating system images across different distributions.

## Architecture

<!-- AI:start:architecture -->
The `oa-tools` project consists of two primary components: `oa` and `coa`. `oa` is a C-based workhorse, while `coa` is a Go-based orchestration tool. These components interact to provide a framework for remastering systems across different Linux distributions. The `Makefile` orchestrates the build process, ensuring that both components are compiled and versioned consistently. Documentation and shell completions are generated by `coa` during the build process.

The repository is structured as follows:

```plaintext
.
├── .github/             # GitHub workflows and CI configurations
├── coa/                 # Source code and docs for the `coa` component
│   ├── docs/            # Generated documentation and shell completions
│   └── main.go          # Entry point for the `coa` binary
├── oa/                  # Source code for the `oa` component
├── scripts/             # Utility scripts
├── tests/               # Test cases for the project
├── Makefile             # Build and clean targets for the project
├── LICENSE              # License file
├── README.md            # Project overview and usage instructions
└── other files          # Miscellaneous configuration and documentation
```

The `Makefile` defines targets for building (`build_oa`, `build_coa`), cleaning, and generating documentation. The `oa` component is built using a C-specific Makefile, while `coa` is compiled with Go. Both binaries are versioned using Git tags.
<!-- AI:end:architecture -->

## CI

<!-- AI:start:ci -->
The repository uses GitHub Actions for continuous integration and automation. Below are the workflows and their purposes:

- **ci-2001.yml to ci-2012.yml**: Run various CI tasks, including building, testing, and validating the project across different configurations.
- **cleanup-branches.yml**: Automatically delete stale branches.
- **cleanup-pollution.yml**: Remove unnecessary files and artifacts from the repository.
- **mirror-orgs-full.yml**: Mirror repositories from external organizations to the current organization.
- **mirror-artifacts.yml**: Sync build artifacts to external storage.
- **sync-forks.yml**: Synchronize forks with upstream repositories.
- **update-infra-deps.yml**: Update infrastructure dependencies.
- **validate-config.yml**: Validate configuration files for correctness.

Required secrets:
- `GH_TOKEN`: GitHub token for repository access.
- `CI_DEPLOY_KEY`: SSH key for deployment tasks.
- `EXTERNAL_STORAGE_KEY`: Access key for external artifact storage.

Refer to individual workflow files in `.github/workflows/` for specific configurations and triggers.
